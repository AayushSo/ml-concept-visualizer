Analyze the staged file diffs and the full content of every affected file in this
repository, then generate one complete commit message.

Repository context:
- This repo is an educational ML/performance concept visualizer collection.
- Most changes are in standalone HTML visualizer pages organized by topic folders.
- Meaningful impact usually involves interactive behavior, formulas, explanations,
  diagrams, controls, simulation logic, navigation, or learning flow.
- Some updates may touch project index/navigation or documentation.

Generate a commit message that strictly follows Conventional Commits.

---

### Analysis & Context Rules

1. Deep contextual analysis (required):
   Do not rely only on hunks. Read full staged-file content to understand the
   feature intent and system-level impact.

2. Meaningful change focus:
   Include only changes that add/fix/refactor behavior, educational content,
   conceptual correctness, or docs that materially improve understanding.

3. Ignore trivial changes:
   - Pure formatting/whitespace/lint-only edits.
   - Import/order-only or dependency metadata-only edits unless they unlock a
     meaningful feature.
   - Generated/build artifacts and lockfiles.

4. Data acquisition requirement:
   Capture staged diff non-interactively using:
   `git --no-pager diff --staged`
   Also inspect full staged file contents (for example via staged blob reads).

5. Repo-aware interpretation:
   Prefer describing user-visible learning/interaction outcomes, such as:
   - corrected math/derivation logic
   - improved visualizer state handling
   - added controls/annotations/explanations
   - fixed navigation/index discoverability

---

### Output Format (strict)

Return only a single plain-text commit message with these sections and no extra
commentary.

1) Subject line
- Format: `<type>: <description>`
- Allowed types: `feat`, `fix`, `refactor`, `docs`, `style`, `test`, `chore`
- Use imperative present tense.
- Max 250 characters.
- Reflect highest-level purpose.

2) Body (optional, 1-5 lines)
- Explain WHY the change was made (motivation/problem solved).
- Wrap at ~80 characters.
- One blank line after subject.

3) Changes section (required)
- Title exactly: `Changes:`
- Then bullets:
  `- <file_path>: <meaningful functional/structural change>`
- Include only meaningful changes.
- Group logically by module/topic when useful.

4) Footer (optional)
- If needed, include:
  `BREAKING CHANGE: <description>`
- Wrap at ~80 characters.
- Separate from previous section with one blank line.

Do not include markdown fences or analysis notes.
